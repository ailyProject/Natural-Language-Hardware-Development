# 第 5 章：串口调试和问题排查

## 本章要解决什么问题

硬件项目出问题时，新手最常说的一句话是：

```text
它没有反应。
```

但“没有反应”太模糊了。程序有没有运行？按键有没有读到？传感器有没有返回数据？LED 引脚有没有被设置？这些都需要证据。

串口调试就是给程序加一张嘴，让开发板把内部状态说出来。

> 图片占位 5-1：开发板通过 USB 串口向电脑输出日志的示意图。

## 5.1 串口是什么

在 Arduino 入门里，串口最常用来输出调试信息。程序里写：

```cpp
Serial.println("LED ON");
```

电脑上的串口工具就能看到 `LED ON`。

这比只看 LED 是否亮更可靠，因为你可以知道程序执行到了哪里，变量值是多少，传感器读数有没有变化。

## 5.2 波特率要一致

串口通信需要设置波特率。程序里常见写法：

```cpp
Serial.begin(9600);
```

那串口工具里也要选择 `9600`。如果程序是 9600，工具选了 115200，就可能看到乱码。

## 5.3 用自然语言让 AI 增加调试输出

很多时候，AI 第一次生成的程序能跑，但调试信息不够。你可以继续说：

```text
请在当前程序中增加串口调试输出。启动时输出“SYSTEM START”，每次读取按键时输出按键状态，每次 LED 状态变化时输出 LED 当前状态。请不要改变原来的控制逻辑。
```

这里的“不要改变原来的控制逻辑”很重要，能减少 AI 过度修改。

如果你只是想让 AI 解释串口日志或分析现象，可以使用问答模式。若希望 AI 直接修改当前项目、增加串口输出或修复程序，就切换到代理模式。代理模式具备项目读写能力，更适合自动编程和错误修复。

## 5.4 调试日志应该输出什么

好的调试日志不只是输出“hello”。它应该回答问题。

| 想确认的问题 | 应输出的内容 |
| --- | --- |
| 程序有没有启动 | `SYSTEM START` |
| 按键有没有读到 | `button=LOW` 或 `button=HIGH` |
| LED 状态有没有改变 | `LED ON`、`LED OFF` |
| 传感器读数是多少 | `temperature=26.5` |
| 程序进入了哪个模式 | `mode=auto` |
| 是否发生异常 | `sensor read failed` |

## 5.5 实验项目：给按键控制 LED 增加日志

### 提示词

```text
请在当前按键控制 LED 项目中加入串口调试信息。

要求：
1. 程序启动时输出“SYSTEM START”。
2. 每次按键状态变化时输出“PRESSED”或“RELEASED”。
3. 每次 LED 状态变化时输出“LED ON”或“LED OFF”。
4. 串口波特率使用 9600。
5. 不要改变原来的按键控制逻辑。

请说明需要增加哪些 Blockly 积木，并解释对应代码思路。
```

### 代码思路

```cpp
void setup() {
  Serial.begin(9600);
  Serial.println("SYSTEM START");
}
```

在状态变化的位置输出：

```cpp
Serial.println("PRESSED");
Serial.println("LED ON");
```

注意，不建议在非常快的循环里每次都输出大量日志，否则串口会刷屏，程序也会变慢。

## 5.6 用日志排查问题

假设 LED 没亮，但串口输出正常，你可以按下面思路判断：

1. 串口有 `SYSTEM START`：说明程序启动了。
2. 串口有按键状态：说明按键输入读到了。
3. 串口有 `LED ON`：说明程序逻辑认为应该点亮 LED。
4. LED 仍然不亮：问题可能在引脚、接线、LED 极性或电阻。

如果串口没有任何输出，则优先检查：

- 程序是否上传成功。
- 串口端口是否选对。
- 波特率是否一致。
- 是否打开了正确的串口工具。
- 程序是否卡在启动前。

## 5.7 把现象说给 AI 听

排查问题时，不要只说“坏了”。可以这样描述：

```text
我的 Arduino Uno 按键控制 LED 项目上传成功了。串口能看到 SYSTEM START，也能看到 PRESSED 和 RELEASED，但 LED 没有变化。LED 使用 D8 外接，接法是 D8 -> 220 欧姆电阻 -> LED 长脚，LED 短脚 -> GND。请帮我分析可能原因，并按检查顺序列出来。
```

这段话给了 AI 足够多的证据，AI 就能更像有经验的同伴一样帮你排查。

编译或上传失败时，不必手动整理所有日志。可以先点击错误详情查看完整输出，然后双击日志中的关键报错，把它提交到 AI 输入框。发送前最好再补一句当前项目目标和你刚刚做过的操作，例如“我正在给按键控制 LED 项目增加串口日志，刚才点击编译后出现这个报错”。这样 AI 更容易判断是库、语法、开发板配置还是端口问题。

## 5.8 常见问题和排查顺序

### 串口没有输出

1. 检查程序是否有 `Serial.begin()`。
2. 检查波特率是否一致。
3. 检查是否选对端口。
4. 检查程序是否上传成功。
5. 检查 USB 数据线。

### 串口乱码

优先检查波特率。程序和串口工具必须一致。

### 输出太快看不清

增加延时，或只在状态变化时输出。

### 输出内容和硬件现象不一致

说明程序逻辑可能正确，但接线或硬件有问题。反过来，如果硬件有动作但日志不对，可能是日志位置写错。

## 5.9 本章练习

1. 给 LED 闪烁程序增加启动日志和状态日志。
2. 给按键项目增加按键计数输出。
3. 故意把串口工具波特率选错，观察乱码。
4. 写一段问题描述，让 AI 根据串口现象给出排查清单。

## 5.10 本章小结

串口调试是嵌入式入门最重要的习惯之一。只要程序会“说话”，你就不再只能猜问题在哪里。

自然语言开发里，串口日志还有一个额外好处：你可以把日志和现象一起发给 AI，让它根据证据帮你分析，而不是凭空猜。
