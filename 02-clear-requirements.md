# 第 2 章：把需求说清楚，再生成程序

## 本章要解决什么问题

很多人第一次用 AI 做硬件项目时，会直接输入一句很短的话：

```text
帮我做一个按键控制灯。
```

这句话没有错，但它太模糊了。AI 不知道你用什么开发板，LED 接在哪个引脚，按键接在哪个引脚，按下后是点亮还是切换状态，也不知道你希望串口输出什么调试信息。

模糊需求并不是不能用。它适合让 AI 先做项目分析，例如“我要开发一个环境数据记录器，帮我设计”。这时你要的是方案、模块选择和任务拆分。可是一旦你希望 AI 直接自动编程，就需要把开发板、模块、引脚、运行规则和验证方法说清楚。

本章的目标很简单：学会把一个模糊想法，改写成 AI 能理解、你能检查、硬件能执行的任务单。

## 2.1 好需求不是写得长，而是关键条件清楚

硬件项目里，一个好需求通常包含五类信息：

1. 我用什么硬件。
2. 我希望检测什么输入。
3. 我希望控制什么输出。
4. 程序按什么规则运行。
5. 我怎么判断它成功了。

传统学习里，通常会先画流程图、写实验目的、列元件清单。自然语言开发并不是把这些省掉，而是把它们换成更容易表达的任务描述。

例如，“按键控制灯”可以改成这样：

```text
我使用 Arduino Uno 做一个按键控制 LED 的项目。

硬件连接：
1. 使用板载 LED，优先使用 LED_BUILTIN。
2. 按键连接到 D2 引脚，使用内部上拉输入。
3. 按键另一端接 GND。

程序行为：
1. 按键没有按下时，LED 熄灭。
2. 按键按下时，LED 点亮。
3. 串口波特率为 9600。
4. 按下时输出“BUTTON PRESSED”，松开时输出“BUTTON RELEASED”。

请生成 Blockly 程序逻辑，并说明对应的 Arduino/C++ 代码思路。最后告诉我编译、上传和测试步骤。
```

这段话并不难，但它把 AI 需要猜的地方都说出来了。

## 2.2 需求拆解表

练习时，你可以先填表，再把表格改写成自然语言。这样能保留传统学习里的“分析过程”。

| 项目项 | 要填写的内容 | 示例 |
| --- | --- | --- |
| 项目目标 | 最终想看到什么现象 | 按下按键 LED 亮，松开 LED 灭 |
| 开发板 | 使用哪块板 | Arduino Uno |
| 输入模块 | 谁负责输入 | 按键，接 D2 |
| 输出模块 | 谁负责输出 | 板载 LED |
| 运行规则 | 输入变化后怎么处理 | 按下亮，松开灭 |
| 调试信息 | 串口输出什么 | 输出按键状态 |
| 验证方法 | 怎么判断成功 | LED 与按键同步变化，串口状态正确 |

## 2.3 用自然语言生成程序前，先问自己六个问题

把需求发给 AI 前，可以先检查这六个问题：

1. 我有没有说清楚开发板型号？
2. 我有没有说清楚输入模块和输出模块？
3. 我有没有说清楚每个模块接到哪个引脚？
4. 我有没有说清楚程序什么时候做什么？
5. 我有没有要求串口输出调试信息？
6. 我有没有要求 AI 给出测试和排查步骤？

如果这六个问题都能回答，AI 生成的结果通常会稳定很多。

## 2.4 把 LLM 最佳实践变成硬件任务单

主流 LLM 实践虽然面向很多不同场景，但放到硬件开发里，可以归纳成几条很实用的规则：

- 指令要具体：不要只说“帮我做一个项目”，要说明目标、硬件、行为和成功标准。
- 上下文要充分：把开发板型号、模块型号、引脚、供电方式、库版本、已有程序和报错日志放进任务里。
- 约束要明确：告诉 AI 哪些东西不能改，哪些安全规则必须保留，哪些输出必须包含。
- 示例能提高稳定性：给一个你喜欢的提示词格式、表格格式或串口日志格式，AI 更容易照着生成。
- 复杂任务要拆开：先让 AI 规划，再让它一步一步执行，不要一次要求它完成所有功能。
- 结果必须验证：把编译结果、上传结果、串口日志和真实硬件现象反馈给 AI，再决定下一步。

你可以把一个硬件提示词写成下面这种结构：

```text
目标：
我想看到什么硬件现象？

上下文：
1. 开发板：
2. 模块：
3. 引脚：
4. 供电方式：
5. 当前程序、库或报错：

约束：
1. 哪些功能必须保留？
2. 哪些引脚或库不要改？
3. 有哪些安全红线？

希望输出：
1. 任务拆解
2. 接线检查清单
3. Blockly 程序逻辑
4. Arduino/C++ 代码思路
5. 编译、上传和测试步骤

验证方式：
我会用什么现象、串口日志或测试表判断成功？
```

这个结构看起来比一句话长，但它能减少 AI 猜测。硬件项目里，少猜一点，就少很多排查时间。

## 2.5 短程任务和长程任务

自然语言开发并不是所有任务都用同一种问法。你需要先判断任务是短程任务还是长程任务。

短程任务目标很小，通常可以一次生成、一次验证。比如：

- 让板载 LED 每 1 秒闪烁。
- 给当前程序增加串口输出。
- 把报警阈值从 30 度改成 28 度。
- 解释一个传感器库的最小示例。
- 根据一段编译错误判断缺少哪个库。

长程任务目标更大，通常需要多个阶段、多次验证和更多上下文。比如：

- 做一个带传感器、OLED、报警和异常处理的环境监测站。
- 做一个有按键短按、长按、状态机和执行器保护的交互作品。
- 做一个联网设备，包含 WiFi 连接、数据上传、断网重连和配置保存。

可以用下面这张表判断：

| 判断问题 | 更像短程任务 | 更像长程任务 |
| --- | --- | --- |
| 目标数量 | 一个明确目标 | 多个功能目标 |
| 涉及模块 | 一个模块或一个小改动 | 多个模块、库或文件 |
| 验证方式 | 一个现象或一条日志 | 多个测试项和阶段验收 |
| AI 输出 | 程序片段、解释、排查建议 | 计划、任务清单、分步实现、测试表 |
| 风险 | 改错后容易回退 | 可能引入接线、电源、依赖或状态逻辑问题 |

短程任务的提示词要直接，重点是说清目标和验证方式。长程任务的提示词要先要求计划，重点是控制范围和阶段验收。

## 2.6 长程任务的自然语言开发方法

长程任务不要从“请直接生成完整项目”开始。更可靠的方法是把一个大目标变成一串短程任务，并且每一步都有验收标准。本书建议使用下面的六步法。

### 第一步：目标定界

先说清最终作品是什么、必须完成哪些功能、哪些内容暂时不做、有哪些安全红线。

```text
我想做一个环境监测站。最终要读取温湿度和光照，OLED 显示数据，温度过高或光照过低时报警。第一版暂时不联网，不保存历史数据。请先帮我确认项目边界、硬件风险和成功标准，不要直接生成程序。
```

### 第二步：上下文装箱

把 AI 需要知道的信息装进一个“上下文包”：开发板、模块、引脚、供电、库、已有程序、错误日志、测试现象。上下文越长，越要用标题、列表和表格分开。

```text
根据下面的上下文帮我继续设计项目。

开发板：Arduino Uno
模块：DHT20、I2C OLED、光敏传感器、报警 LED
引脚：DHT20 和 OLED 共用 I2C，光敏 AO 接 A0，报警 LED 接 D9
供电：全部使用开发板 5V，LED 串联 220 欧姆电阻
已完成：板载 LED 闪烁和串口输出已经验证通过
```

### 第三步：让 AI 先规划

长程任务开始时，先让 AI 输出阶段计划、依赖关系、每阶段验证方法和风险点。此时不要急着要完整代码。

```text
请把这个项目拆成 5 到 7 个阶段。每个阶段请列出目标、需要修改的内容、验证方法和可能风险。先不要生成完整程序。
```

### 第四步：一次只执行一个短程任务

从计划中选最小的一步执行。例如先只读传感器，再加显示，再加报警。每次提示词都要说明“基于当前项目”“只修改必要部分”“保留已经验证通过的功能”。

```text
基于当前项目，请只完成第一阶段：每 2 秒读取温湿度并输出到串口。暂时不要加入 OLED、光敏传感器和报警 LED。请说明 Blockly 逻辑、代码思路和测试步骤。
```

### 第五步：用真实结果验收

每一步完成后，都要编译、上传、看串口和硬件现象。把结果原样反馈给 AI，而不是只说“失败了”。

```text
上传成功，但串口输出一直是 SENSOR ERROR。DHT20 接线是 SDA 接 A4，SCL 接 A5，电源接 5V，GND 已连接。请按接线、I2C 地址、库初始化、传感器故障四个方向给我排查顺序。
```

### 第六步：记录状态，再进入下一步

长程任务最怕“上下文断掉”。每完成一个阶段，都记录当前版本做到了什么、验证结果是什么、还有哪些问题。下一次让 AI 修改时，把这段记录放进去。

```text
当前版本记录：
1. 温湿度读取已经通过，串口每 2 秒输出一次。
2. 光敏传感器还没有加入。
3. OLED 还没有加入。
4. 已知问题：第一次上电时偶尔读取失败一次，但之后恢复正常。

请基于这个状态，继续加入 OLED 显示。不要改动已经通过的温湿度读取逻辑。
```

这套方法的核心不是让提示词变得复杂，而是让每一次 AI 协作都有边界、有上下文、有验收。长程任务越复杂，越要把 AI 当成协作开发者，而不是一次性答案生成器。

## 2.7 实验项目：把“按键控制灯”改成可执行任务

### 第一步：写出原始想法

```text
我想做一个按键控制灯。
```

### 第二步：补充硬件条件

```text
我使用 Arduino Uno。LED 先使用板载 LED。按键接到 D2 引脚，使用内部上拉，按键另一端接 GND。
```

### 第三步：补充程序规则

```text
按键按下时 LED 点亮，松开时 LED 熄灭。串口输出当前按键状态和 LED 状态。
```

### 第四步：补充输出要求

```text
请生成 Blockly 程序逻辑，解释对应 Arduino/C++ 代码，并给出接线检查、编译上传和测试步骤。
```

### 第五步：合并成完整提示词

```text
我使用 Arduino Uno 做一个按键控制 LED 的项目。

硬件连接：
1. LED 使用板载 LED，也就是 LED_BUILTIN。
2. 按键连接到 D2 引脚。
3. 按键使用内部上拉输入，按键另一端接 GND。

程序行为：
1. 按键按下时，LED 点亮。
2. 按键松开时，LED 熄灭。
3. 串口波特率为 9600。
4. 按键状态变化时，在串口输出当前状态。

请输出：
1. Blockly 程序逻辑。
2. 对应 Arduino/C++ 代码思路。
3. 接线检查清单。
4. 编译、上传和测试步骤。
5. 如果 LED 不亮或串口无输出，应按什么顺序排查。
```

## 2.8 AI 生成结果后怎么检查

AI 回复后，不要只看有没有代码。按下面顺序检查：

1. 是否使用了 Arduino Uno。
2. 是否使用了 `LED_BUILTIN` 或正确的 LED 引脚。
3. 按键是否接到 D2。
4. 是否启用了内部上拉。
5. 是否说明了“内部上拉时，按下通常读到 LOW”。
6. 是否有串口输出。
7. 是否给了测试步骤。

这里特别注意第 5 条。很多新手会以为按下就是 HIGH，但使用内部上拉时，按键按下通常会把引脚接到 GND，所以读数是 LOW。

## 2.9 Blockly 和代码怎么对应

这个项目的程序结构大概是：

1. 启动串口。
2. 设置 LED 为输出。
3. 设置按键为内部上拉输入。
4. 循环读取按键状态。
5. 如果按下，点亮 LED。
6. 如果松开，熄灭 LED。
7. 输出状态到串口。

对应 Arduino/C++ 思路如下：

```cpp
const int buttonPin = 2;

void setup() {
  Serial.begin(9600);
  pinMode(LED_BUILTIN, OUTPUT);
  pinMode(buttonPin, INPUT_PULLUP);
}

void loop() {
  int buttonState = digitalRead(buttonPin);

  if (buttonState == LOW) {
    digitalWrite(LED_BUILTIN, HIGH);
    Serial.println("BUTTON PRESSED, LED ON");
  } else {
    digitalWrite(LED_BUILTIN, LOW);
    Serial.println("BUTTON RELEASED, LED OFF");
  }

  delay(100);
}
```

这段代码可以帮助你理解 Blockly 积木背后的传统程序结构。

## 2.10 常见问题

### LED 一直亮

可能原因：按键接线相反、程序判断条件写反、引脚悬空。

可以问 AI：

```text
我的按键控制 LED 项目中，LED 一直亮。按键使用 D2 和内部上拉，另一端接 GND。请帮我按接线、程序判断、串口输出三个方向排查。
```

### 按键没有反应

可能原因：按键没有接到 GND、引脚选错、没有启用内部上拉、开发板型号不对。

### 串口刷屏太快

可以增加状态变化检测，只在按下或松开瞬间输出，也可以先把 `delay()` 调大。

## 2.11 本章练习

1. 把“按键控制灯”改成“按一下切换一次 LED 状态”。
2. 增加一个计数器，统计按键被按下了多少次。
3. 把完整需求写成任务单，再让 AI 生成程序。
4. 对比“模糊需求”和“清晰需求”得到的 AI 回复差异。
5. 判断下面三个任务分别是短程任务还是长程任务：修改 LED 闪烁间隔、制作环境监测站、把一个传感器库转换成 Blockly 库。
6. 选择一个长程任务，按“目标定界、上下文装箱、先规划、单步执行、真实验收、记录状态”写出第一轮提示词。

## 2.12 本章小结

这一章最重要的不是按键，也不是 LED，而是需求表达和任务管理。以后做任何硬件项目，都先问自己：硬件是什么、输入是什么、输出是什么、规则是什么、怎么验证。

自然语言开发的能力，不只是“会问 AI”，而是“会把工程问题说清楚”。短程任务要清楚直接，长程任务要先规划、再拆分、逐步验收。这样 AI 才能真正成为硬件开发过程中的协作助手。
