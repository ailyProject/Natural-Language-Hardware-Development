# 第 14 章：综合项目二：可交互控制作品

## 本章要解决什么问题

上一章的环境监测站主要是“读取和显示”。这一章我们做一个更有交互感的作品：有输入、有输出、有状态、有反馈。

可交互控制作品可以有很多形式，比如桌面提醒器、智能灯控、简易小车、自动浇水模型。你不一定只能选择一种作品，但方法应该一致：先定义状态，再定义触发条件，再定义输出动作。

## 14.1 什么是状态机

很多交互项目不是简单的“按下就亮”。它可能有多个状态：

- 待机。
- 工作中。
- 报警。
- 暂停。
- 错误。

状态机会记录当前处于哪种状态，并根据输入切换到新状态。

可以把它说成：

```text
现在是什么状态？发生了什么事件？接下来变成什么状态？要做什么动作？
```

## 14.2 用自然语言描述交互项目

交互项目提示词要说清楚：

1. 有哪些输入。
2. 有哪些输出。
3. 有哪些状态。
4. 什么事件会切换状态。
5. 每个状态下显示什么、输出什么。
6. 异常时如何处理。

示例：

```text
我想做一个桌面提醒器。有一个按键、一个蜂鸣器、一个 OLED。短按按键开始计时，再短按暂停，长按清零。计时结束时蜂鸣器响三次，OLED 显示 DONE。请先帮我设计状态机，再生成 Blockly 程序。
```

### 交互项目的长程开发顺序

交互项目很容易因为状态多、触发条件多而混乱。建议把它当成长程任务处理，按下面顺序推进：

1. 先列输入、输出和状态，不生成程序。
2. 再画状态表，确认每个事件会让状态怎么变化。
3. 只实现输入识别，例如短按、长按和消抖，并用串口验证。
4. 再实现状态切换，继续用串口输出状态变化。
5. 最后接入蜂鸣器、OLED、电机或水泵等输出动作。
6. 增加异常处理和停止条件，特别是执行器保护。

可以这样要求 AI：

```text
这是一个长程交互项目。请先只完成状态设计，不要生成完整程序。请输出输入清单、输出清单、状态清单、事件清单和状态转换表，并指出哪些地方需要消抖、长按判断或执行器保护。
```

等状态表确认后，再让 AI 逐步实现。这样可以避免一开始就得到一大段难以检查的程序。

## 14.3 项目方案一：桌面提醒器

### 功能描述

- OLED 显示当前状态和剩余时间。
- 按键短按开始或暂停。
- 按键长按清零。
- 时间到蜂鸣器响三次。

### 提示词

```text
我使用 Arduino Uno 做一个桌面提醒器。

硬件连接：
1. 按键接 D2，使用内部上拉。
2. 有源蜂鸣器接 D9。
3. OLED 使用 I2C，SDA 接 A4，SCL 接 A5。

功能要求：
1. 有三个状态：IDLE、RUNNING、PAUSED。
2. 短按按键：IDLE 进入 RUNNING，RUNNING 进入 PAUSED，PAUSED 回到 RUNNING。
3. 长按按键超过 2 秒：回到 IDLE，并清零计时。
4. 计时到 60 秒时，蜂鸣器响三次，OLED 显示 DONE。
5. 串口输出状态变化。

请先画出状态机表，再生成 Blockly 程序逻辑和 Arduino/C++ 代码思路。
```

## 14.4 项目方案二：智能灯控

智能灯控适合结合光敏传感器、按键和 LED。

功能示例：

- 自动模式：环境暗时开灯，亮时关灯。
- 手动模式：按键切换灯状态。
- 长按按键切换自动/手动模式。
- OLED 或串口显示当前模式。

提示词示例：

```text
我想做一个智能灯控项目。光敏传感器接 A0，LED 接 D9，按键接 D2 并使用内部上拉。项目有 AUTO 和 MANUAL 两个模式。AUTO 模式下光线低于阈值自动开灯，MANUAL 模式下短按按键切换 LED 状态，长按按键切换模式。请先设计状态和触发规则，再生成 Blockly 程序。
```

## 14.5 项目方案三：自动浇水模型

自动浇水模型适合讲传感器、执行器和保护逻辑。

功能示例：

- 土壤湿度低于阈值时打开水泵。
- 水泵最多连续运行 5 秒。
- 按键可以手动停止。
- 串口输出湿度、水泵状态和保护状态。

提示词示例：

```text
我想做一个自动浇水模型。土壤湿度传感器 AO 接 A0，水泵通过安全的低压驱动模块控制，控制引脚接 D6，按键接 D2 用作停止按钮。湿度低于阈值时启动水泵，但水泵连续运行不能超过 5 秒。请先列出电源安全注意事项，再设计程序逻辑。
```

注意：水泵项目必须先确认供电和驱动方案，不建议直接把水泵接到开发板引脚上。

## 14.6 状态表比长代码更重要

交互项目容易乱，通常是因为没有先想清状态。建议先填状态表：

| 当前状态 | 事件 | 下一个状态 | 输出动作 |
| --- | --- | --- | --- |
| IDLE | 短按 | RUNNING | 开始计时 |
| RUNNING | 短按 | PAUSED | 暂停计时 |
| PAUSED | 短按 | RUNNING | 继续计时 |
| 任意状态 | 长按 | IDLE | 清零并停止输出 |

把状态表交给 AI，比直接说“做一个复杂按钮功能”稳定很多。

## 14.7 测试清单

- 每个状态是否能进入。
- 每个按键动作是否只触发一次。
- 长按和短按是否区分正确。
- 输出动作是否和状态一致。
- 串口是否记录状态变化。
- 异常或停止条件是否有效。
- 断电重启后状态是否符合预期。

## 14.8 常见问题

### 按一次跳过多个状态

可能是没有消抖，或程序在长按时持续触发。

### 状态显示和实际动作不一致

可能是显示逻辑和控制逻辑分开修改后不同步。让 AI 按状态表统一检查。

### 执行器停不下来

说明保护逻辑不足。要增加最大运行时间、急停按键或异常关闭。

## 14.9 本章练习

1. 为桌面提醒器画状态机。
2. 把智能灯控项目拆成 AUTO 和 MANUAL 两个模式。
3. 给自动浇水模型增加手动停止按钮。
4. 让 AI 根据状态表生成 Blockly 程序。
5. 写一份项目测试清单。

## 14.10 本章小结

可交互项目的核心不是模块多，而是状态清楚。先定义状态和事件，再让 AI 生成程序，会比直接要求它写完整项目更可靠。

自然语言在这里最适合做状态描述、任务拆分和测试清单。你要学会控制复杂度，一次只验证一个状态变化。
